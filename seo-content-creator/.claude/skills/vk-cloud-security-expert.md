### Zero Trust для надёжности

VK Cloud применяет принципы Zero Trust не только для безопасности, но и для обеспечения максимальной надёжности и отказоустойчивости платформы.

```yaml
zero_trust_reliability:
  principle: "Никому не верь — даже собственной инфраструктуре"
  implementation:
    - Регулярные испытания на отказоустойчивость
    - Stage-окружение идентичное production
    - Физические тесты отключения зон доступности
    - Проверка всех регламентов на практике

tested_scenarios:
  - "12 испытаний на Stage за полгода (раз в 2 недели)"
  - "30 мая 2025: физическое отключение control plane зоны GZ на проде"
  - "Подтверждена работоспособность при отказе одной AZ"
```

## WRITING GUIDELINES

### Терминология

**Используй правильную терминологию:**

| Правильно | Неправильно |
|-----------|-------------|
| информационная безопасность | инфобез, кибербез (в тексте) |
| уязвимость | дыра, баг безопасности |
| злоумышленник, атакующий | хакер (нейтральный контекст) |
| инцидент безопасности | взлом (если не подтверждён) |
| аутентификация | авторизация (когда речь о проверке личности) |
| авторизация | аутентификация (когда речь о правах доступа) |
| шифрование | кодирование (разные вещи) |
| персональные данные | личные данные (в контексте 152-ФЗ) |

### Стиль написания

```yaml
tone:
  - Профессиональный, но доступный
  - Без запугивания читателя
  - С конкретными рекомендациями
  - Основанный на фактах и стандартах

structure:
  - Начинать с проблемы/риска
  - Объяснять последствия
  - Давать практические рекомендации
  - Ссылаться на стандарты и best practices

avoid:
  - Преувеличение угроз (FUD)
  - Технический жаргон без объяснения
  - Устаревшие рекомендации
  - Абсолютные утверждения ("100% защита")
```

## SECURITY STRATEGIES

### Zero Trust Architecture (ZTA)

```yaml
zero_trust:
  definition: >
    Zero Trust — это не продукт и не технология, а концепция архитектуры,
    меняющая сам принцип построения цифровых систем.

  core_principle: >
    «Не доверяй никому и ничему по умолчанию. Всегда проверяй.
    Давай минимально необходимый доступ — только на время
    и только тому, кому действительно нужно.»

  origin:
    author: "Джон Киндерваг (Forrester)"
    year: 2010
    standardization: "NIST SP 800-207 «Zero Trust Architecture»"
    status: "Де-факто международный стандарт"

  adoption: "~30% российских компаний используют Zero Trust"

why_old_model_failed:
  problem: "Крах модели 'безопасного периметра'"
  reasons:
    - "Бизнес-приложения переехали в облака"
    - "Сотрудники работают из дома, командировок, через личные устройства"
    - "Данные в десятках систем и микросервисов"
    - "Злоумышленники атакуют через доверенные каналы и учётные записи"
    - "40%+ атак — через компрометацию учётных записей (BI.ZONE, Positive Technologies)"

four_principles:
  1_no_implicit_trust:
    principle: "Нельзя доверять по умолчанию"
    explanation: >
      Пользователь, устройство или процесс не получают доверия
      только потому, что находятся «внутри сети».

  2_dynamic_verification:
    principle: "Динамическая проверка каждого запроса"
    explanation: >
      Каждый запрос доступа проверяется с учётом контекста:
      личность, устройство, местоположение, уровень риска.

  3_least_privilege:
    principle: "Минимально возможный и временный доступ"
    explanation: >
      Доступ выдаётся только на время конкретной операции
      и только к необходимым ресурсам.

  4_continuous_monitoring:
    principle: "Непрерывный контроль и корректировка"
    explanation: >
      Системы безопасности постоянно контролируют поведение
      и автоматически корректируют политику доступа.
```
